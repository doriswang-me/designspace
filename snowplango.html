01 / The Problem

Skiers don't need more weather data.
They need an answer.

Existing tools dump barometric pressure, multi-model forecasts and avalanche risk on you and walk away. The skier still has to translate "8 inches overnight, 18 mph wind, –6°C base" into a real decision: is this resort, this weekend, going to be fun for me?

01

Forecasts treat every skier the same.

A powder hunter and a park rider open the exact same weather screen. Same numbers, same charts — but a 14 inch dump means very different things to each of them.

02

Trip planning lives in twelve tabs.

Users compare resorts across snow apps, IKON/Epic pass coverage, Google Maps, Booking.com, Reddit threads and group chats — then guess. There's no place to hold the whole decision.

03

Conditions change hour-to-hour.

By 11 AM the morning powder is tracked out and the south-facing groomers are ice. A resort-level daily forecast can't see that. Trail-level intelligence can.

02 / Competitive Landscape

OpenSnow gives you the weather.
Snowplango gives you the answer.

OpenSnow is the category leader — and a brilliant meteorology product. But it's built for skiers who already speak fluent forecast. We picked a different lane: turn raw data into a personalised, ranked recommendation that anyone can act on.

05 / Design System

Built for an alpine night.
Quiet, cold, with the sky doing the work.

Snowplango opens before sunrise — in lift lines, on freezing fingers, against bright snow glare. The system leans into a deep navy canvas with restrained accents, a slow aurora gradient for atmosphere, and four scoring colors that map one-to-one to the riding-style dimensions everywhere they appear in the product.

Motion · Principles
  • Aurora breathes.
    Background opacity drifts 0.3 → 0.5 → 0.3 over a slow 12s loop — atmosphere, not decoration.
  • Numbers count up.
    Score values animate 0 → target over 0.8s, staggered 80ms across the four dimensions, so the eye reads them in sequence.
  • AI types like a person.
    Tool-use lines stream in italic gray; responses stream token-by-token at 15–25ms — borrowed honestly from ChatGPT & Claude conventions.
  • Reorder is a spring.
    When the user changes filter, resort rows resort with a 0.6s spring (300/30) — never a hard reload.
Iconography · Weather System

Eight conditions, one visual language.

Custom-built rounded weather glyphs designed for the dark navy product surface. Each icon uses the same stroke weight, the same cloud silhouette, and the four-color palette (ice white · gold sun · powder blue · accent yellow) so the home screen reads as one system instead of an icon-pack mash-up.

07 / My Role

Lead designer,
paired with one engineer.

A two-person team — me on every design surface, one engineer on iOS & the data pipeline. Tight loop, weekly ship cadence, design and code reviewed in the same Figma file.

What I owned

  • Competitive teardowns of OpenSnow, OnTheSnow & Snow-Forecast — and the positioning argument that came out of them.
  • The four-dimension scoring model as a UX construct — naming, color tokens, copy, score-tile component.
  • End-to-end onboarding flow (Splash → Welcome → Scores → Rankings → AI Planner) including animation specs.
  • Design system foundations: Navy palette, Inter × Instrument Serif type pairing, motion principles, component library.
  • Custom weather icon library — eight conditions in one visual language.
  • AI conversation design — tool-use status lines, streaming bubbles, error states, escape hatches.
  • Brand & marketing surfaces: landing page covers (desktop & mobile), App Store assets, copywriting.
  • Specs & engineering handoff for v1.0; current design lead on the trail-forecasting beta.

How we worked together

  • Shared Figma file as the single source of truth — engineer commented directly on frames instead of separate tickets.
  • I prototyped score-tile interactions in code so we could agree on motion timing before hand-off, not after.
  • Weekly review of what shipped, what slipped, what the score model got wrong — copy and visual updates driven by real data.
  • Trade-offs negotiated openly: when the trail-forecast model couldn't hit hourly accuracy, we co-designed a "morning / midday / afternoon" UI instead of waiting.
